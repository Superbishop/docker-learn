📦 什么是 Docker？

Docker 是一个容器化平台，让你把应用和它所需的一切（代码、运行时、系统工具、库）打包成一个轻量、可移植的容器，在任何机器上都能一致运行。

🏗️ 核心概念：虚拟机 vs 容器

对比项🖥️ 虚拟机 (VM)📦 Docker 容器
启动速度分钟级秒级 ⚡
资源占用完整操作系统，GB级共享内核，MB级
隔离性完全隔离进程级隔离
可移植性较好极好 ✅
性能有损耗接近原生
大小GB ~ TBMB ~ GB

🧱 Docker 三要素

🖼️

Image（镜像）

容器的"模板"——只读蓝图，包含程序+依赖+配置。相当于面向对象的类。

📦

Container（容器）

镜像的运行实例——一个活着的、独立的进程。相当于面向对象的对象。

📚

Repository（仓库）

存放镜像的地方。Docker Hub 是最大的公共仓库，类似 GitHub。

🛠️ 核心命令

Docker 命令浩如烟海，但 80% 的场景只用到下面这些：

📥 拉取 & 运行镜像

BASH
# 拉取镜像（从 Docker Hub）
docker pull nginx:latest

# 运行一个容器（交互式）
docker run -it ubuntu:22.04 bash

# 运行一个容器（守护进程/后台）
docker run -d --name my-nginx nginx:latest

# 运行并映射端口（宿主机:容器）
docker run -d -p 8080:80 nginx:latest
#  → 访问 http://localhost:8080 即可看到 nginx 页面

# 运行并挂载数据卷
docker run -d -v /my/data:/data --name my-app my-image

# 覆盖容器启动命令
docker run -d my-image python app.py

🔍 查看 & 管理

BASH
# 列出运行中的容器
docker ps

# 列出所有容器（包括已停止）
docker ps -a

# 查看镜像列表
docker images

# 查看容器日志
docker logs my-nginx
docker logs -f my-nginx   # 实时跟踪
docker logs --tail 100 my-nginx  # 最后100行

# 查看容器内进程
docker top my-nginx

# 检查容器详情
docker inspect my-nginx

⏯️ 启停 & 删除

BASH
# 启动已停止的容器
docker start my-nginx

# 停止容器（优雅停止）
docker stop my-nginx

# 强制停止
docker kill my-nginx

# 重启容器
docker restart my-nginx

# 删除容器（必须先停止）
docker rm my-nginx

# 强制删除运行中的容器
docker rm -f my-nginx

# 删除镜像
docker rmi nginx:latest

🔧 进入容器 & 执行命令

BASH
# 在运行中的容器内执行命令
docker exec my-nginx ls /usr/share/nginx/html

# 以 root 用户进入容器（调试用）
docker exec -u root -it my-nginx bash

# 复制文件（容器 ↔ 宿主机）
docker cp my-nginx:/etc/nginx/nginx.conf ./nginx.conf
docker cp ./local-file.txt my-nginx:/tmp/

🧹 清理

BASH
# 删除已停止的容器
docker < class="cmd">container prune

# 删除悬空镜像（无标签）
docker image prune

# 删除所有未使用的镜像、容器、网络
docker system prune

# 完全清理（包括数据卷）
docker system prune -a --volumes

📝 Dockerfile — 构建自定义镜像

Dockerfile 是容器的"构建配方"，告诉 Docker 如何从零构建你的自定义镜像。

📋 常用指令

DOCKERFILE
# 基础镜像（必须）：所有指令的起点
FROM python:3.11-slim

# 维护者信息
MAINTAINER bishop@example.com

# 工作目录（后续指令在此目录执行）
WORKDIR /app

# 复制文件（构建上下文 → 镜像内）
COPY requirements.txt .
COPY . .

# 运行命令（构建时执行）
RUN pip install --no-cache-dir -r requirements.txt

# 设置环境变量
ENV NODE_ENV=production
ENV PORT=8080

# 暴露端口（仅文档作用）
EXPOSE 8080

# 设置启动命令（容器运行时执行，可被 docker run 覆盖）
CMD ["python", "app.py"]
# 注意：CMD 只有最后一个生效，格式为 JSON 数组

# 设置入口点（与 CMD 配合使用）
ENTRYPOINT ["python", "app.py"]

@app.route('/')

    return "Hello from Docker! 🐳"

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000)
📦 requirements.txt
flask==3.0.*
DOCKERFILE
FROM python:3.11-slim

# 设置工作目录
WORKDIR /app

# 先复制依赖文件，再安装（利用 Docker 缓存层）
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 再复制应用代码
COPY app.py .

# 暴露端口
EXPOSE 5000

# 启动命令
CMD ["python", "app.py"]
TERMINAL — 构建 & 运行
# 构建镜像（. 表示构建上下文为当前目录）
docker build -t my-flask-app:latest .

# 运行
docker run -d -p 5000:5000 --name flask-app my-flask-app:latest

# 测试
curl http://localhost:5000
# → Hello from Docker! 🐳

🎯 Dockerfile 最佳实践

  • ✅ 用 slim/alpine 镜像 — 减少体积，如 python:3.11-slim 而非 python:3.11
  • ✅ 利用构建缓存 — 先 COPY 依赖文件，再 COPY 代码，修改代码不需重装依赖
  • ✅ 用 .dockerignore — 排除 node_modules、.git、__pycache__ 等
  • ✅ 一个容器一个进程 — 不要在容器里运行多个服务（用 docker-compose）
  • ❌ 别用 ROOT 运行应用 — 用 USER 指令切换到普通用户

🔗 Docker Compose — 多容器编排

真实应用往往需要多个容器（Web + DB + Redis）。Docker Compose 用一个 YAML 文件定义和运行整套应用。

📁 示例：Flask + Redis + Nginx

📁 目录结构
project/
├── docker-compose.yml
├── app/
│   ├── app.py
│   └── requirements.txt
└── nginx/
    └── nginx.conf
YAML — docker-compose.yml
services:
  web:
    build: ./app
    ports:
      - "5000:5000"
    environment:
      - REDIS_HOST=redis
      - REDIS_PORT=6379
    depends_on:
      - redis
    restart: unless-stopped

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    volumes:
      - redis-data:/data
    restart: unless-stopped

  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/conf.d/default.conf:ro
    depends_on:
      - web

volumes:
  redis-data:

🎮 Compose 核心命令

BASH
# 启动所有服务（后台运行）
docker compose up -d

# 带构建（代码变更后重新构建镜像）
docker compose up --build -d

# 查看运行状态
docker compose ps

# 查看日志（所有服务）
docker compose logs -f
docker compose logs -f web   # 只看 web 服务

# 停止并删除（不删除数据卷）
docker compose down

# 完全清理（包括数据卷 —⚠️会丢数据）
docker compose down -v

# 重启服务
docker compose restart web

# 进入服务容器
docker compose exec web bash

💾 数据管理 — 数据卷 & 挂载

容器的文件系统是临时性的——容器删除，数据丢失。Docker 提供两种持久化方案：

🔄 数据卷（Volumes）— 官方推荐

BASH
# 创建命名数据卷
docker volume create my-data

# 运行容器并挂载数据卷
docker run -d -v my-data:/var/lib/postgresql/data --name pg postgres:15

# 或者在 docker-compose.yml 中：
volumes:
  my-data:
    external: true   # 使用已存在的数据卷
services:
  db:
    volumes:
      - my-data:/var/lib/postgresql/data

# 查看所有数据卷
docker volume ls

# 删除未使用的数据卷
docker volume prune

📂 绑定挂载（Bind Mount）— 同步宿主机目录

BASH
# 格式：宿主机路径:容器内路径
docker run -d \
  -v /home/bishop/project:/app \
  -p 3000:3000 \
  --name dev-node \
  node:18

# docker-compose 中：
services:
  web:
    volumes:
      - ./project:/app        # 绑定挂载（代码热更新）
      - /tmp/logs:/logs       # 匿名挂载（临时）

⚡ tmpfs 挂载 — 内存文件系统（敏感数据）

BASH
# 数据存在内存中，容器停止即消失，适合临时/敏感数据
docker run -d \
  --tmpfs /tmp:rw,noexec,nosuid,size=100m \
  --name secure-app my-app

🔍 三种挂载方式对比

类型语法持久化性能适用场景
Volume-v my-vol:/data✅ Docker 管理高数据库、日志
Bind Mount-v /host:/container✅ 宿主机最高开发调试、代码热更新
tmpfs--tmpfs /data❌ 内存中最高敏感临时数据

🌐 Docker 网络

默认情况下，Docker 会创建一个 bridge 网络，所有容器在其中可以互相通信，但与宿主机隔离。

🛣️ 端口映射（让外部访问容器）

BASH
# -p 宿主机端口:容器端口
docker run -d -p 8080:80 nginx   # 宿主机8080 → 容器80
docker run -d -p 443:443 -p 8080:80 nginx   # 多端口
docker run -d -p 127.0.0.1:8080:80 nginx  # 仅本机访问
docker run -d -P nginx              # 随机端口（EXPOSE 定义的）

🔀 Docker 网络模式

🌉

bridge（默认）

所有容器连到虚拟网桥，可通过 -p 暴露端口给宿主机。同一网络的容器可互相通信。

📍

host（共享宿主机网络）

容器直接使用宿主机网络栈，无隔离。性能最好，但端口冲突风险高。

🔗

overlay（跨主机）

连接多个 Docker 守护进程，支持 Swarm 集群跨主机通信。

📴

none（无网络）

容器完全无网络连接，用于完全隔离的作业。

BASH
# 指定网络模式
docker run --network host my-app

# 查看网络列表
docker network ls

# 创建自定义 bridge 网络（推荐）
docker network create --driver bridge my-net

# 容器加入自定义网络（自动 DNS 解析）
docker run -d --network my-net --name web nginx
docker run -d --network my-net --name api my-api
# → web 容器可以用 "api" 主机名直接访问 api 容器

📦 api/server.js
const express = require('express');
const mongoose = require('mongoose');
const app = express();

app.use(express.json());

// 连接 MongoDB（使用环境变量或 docker-compose 定义的主机名）
const MONGO_URI = process.env.MONGO_URI || 'mongodb://localhost:27017/todo';
mongoose.connect(MONGO_URI)
  .then(() => console.log('✅ MongoDB connected'))
  .catch(err => console.error('❌ MongoDB error:', err));

// Todo 模型
const TodoSchema = new mongoose.Schema({
  title: String,
  done: { type: Boolean, default: false },
  createdAt: { type: Date, default: Date.now }
});
const Todo = mongoose.model('Todo', TodoSchema);

// API 路由
app.get('/api/todos', async (req, res) => {
  const todos = await Todo.find().sort({ createdAt: -1 });
  res.json(todos);
});

app.post('/api/todos', async (req, res) => {
  const todo = new Todo({ title: req.body.title });
  await todo.save();
  res.status(201).json(todo);
});

app.delete('/api/todos/:id', async (req, res) => {
  await Todo.findByIdAndDelete(req.params.id);
  res.json({ message: 'Deleted' });
});

const PORT = process.env.PORT || 3000;
app.listen(PORT, () => console.log(`🚀 Server running on :${PORT}`));
